A common pattern is to mount the PDF viewer inside a tab, an Ionic route transition, or any other container that is hidden when the application loads. The viewer is mounted right away but lives inside a display: none subtree until the user activates the tab.

Two historical bugs made this awkward:

  • Before version 22.3.0, the viewer attempted to initialize PDF.js while hidden, failed (because pages have no layout when their ancestor is display: none), and never recovered even after the container was shown — users saw an empty viewer. Fixed in #2749 by a waitForRootElement() guard inside ngx-extended-pdf-viewer that delays initialization until the host element gains an offsetParent. Tab 2 below demonstrates this: the viewer is mounted from the start but only initializes the first time you switch to Tab 2, and [page]="5" takes effect normally.
  • Until version 27.5.0, the corner case of a viewer that initialized visibly and was later hidden (e.g. a route transition fires scrollPageIntoView() mid-flight, or application code calls the scroll service while the viewer is on an inactive tab) still failed silently with offsetParent is not set. #1321 fixed that: such scroll requests are now deferred via ResizeObserver + IntersectionObserver and replayed when the viewer becomes visible again.

If you are running this showcase locally a third tab appears with a regression test for that corner case — it is hidden in the public showcase because the workflow is too involved to be a useful product demo.
