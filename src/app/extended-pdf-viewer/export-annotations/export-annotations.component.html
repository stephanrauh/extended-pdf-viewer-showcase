Understanding Annotation IDs

Every serialized annotation carries two kinds of identifier. Knowing which is which saves a lot of confusion when you store annotations and load them back later.

Field id customId (since v28, #3225)
Who sets it pdf.js (automatic) You (e.g. a UUID)
Survives a JS save/restore round-trip?
(getSerializedAnnotations() → your storage → addEditorAnnotation())		 No — ignored on re-add, a fresh id is assigned Yes — preserved, as long as you persist the JSON and re-apply it
Survives exporting / downloading a PDF file?
(the Download button — writing a .pdf, then reopening it; not "Export annotations", which is JSON)		 No No — it lives only in this JS API, never in the file
Use it for Live event tracking within one session Matching annotations to your own records

The temporary id

id (e.g. pdfjs_internal_editor_3) is handy for real-time tracking: it appears on every (annotationEditorEvent), and you can look an annotation up with getSerializedAnnotation(id). But it is regenerated each session and is dropped when you re-add an annotation, so never persist it.

The stable customId (#3225)

Set your own customId on a serialized annotation before you store it, and it is preserved through the getSerializedAnnotations() → store → addEditorAnnotation() round-trip. It is also surfaced on every (annotationEditorEvent) and is accepted by getSerializedAnnotation(customId), so it is the field to rely on when you need a durable handle. (Keeping customId values unique is your responsibility.)

What "stable" means here: it survives your JavaScript save/restore cycle — you read the annotations out, keep the JSON in your own storage (localStorage, a database, …), and feed it back later with addEditorAnnotation(). It is not stored in the PDF itself. If you hit the Download button and reopen the resulting file, the customId is gone — it never travels into the document (and neither does the temporary id). It lives purely in this JavaScript export/import API.

Best Practice: Storing & Restoring with a Stable ID

// Save: give each annotation your own stable id (keep it if it already has one).
const annotations = pdfService.getSerializedAnnotations() ?? [];
const toStore = annotations.map((a) => ({ ...a, customId: a.customId ?? crypto.randomUUID() }));
localStorage.setItem('annotations', JSON.stringify(toStore));

// Restore: customId is preserved (only the throwaway `id` is regenerated).
const stored = JSON.parse(localStorage.getItem('annotations'));
await pdfService.addEditorAnnotation(stored);

// getSerializedAnnotations() now returns the same customId values you stored.

⚠️ Important: "Persisting" here means keeping the serialized JSON in your own storage and re-applying it with addEditorAnnotation() — use customId for that, since the id you pass in is always ignored. Neither id is written into the PDF, so downloading the file and reopening it does not bring either back.
